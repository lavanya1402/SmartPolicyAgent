Engineering SOP – Internal Use Only

1. Code Repository & Naming Conventions

All repositories follow the format: team-product-purpose. Examples:

  • core-api-authentication
  • frontend-client-dashboard

Repo creation is handled by Engineering Ops. All new services must be registered in AlturaInfraRegistry.

2. CI/CD Strategy

All code pushes must go through the Altura CI pipeline:

  1. Pre-commit hooks via altura-linter
  2. PR-level build & test (Jenkins + GitHub Actions hybrid)
  3. Staging deployment via altura-deployctl
  4. Prod deployments require dual approval from Tech Lead & Security

3. Infrastructure & Environments

We use a 3-tier architecture (Dev → Staging → Prod) with locked down roles:

  • Dev: Auto-deploy every push
  • Staging: Scheduled via altura-deployctl --env staging
  • Production: Requires signed artifact from ReleaseGate

4. Monitoring & On-Call Rotation

All production services must be registered with AlturaWatch for uptime and latency tracking.

On-call assignments rotate weekly. All alerts routed through PagerDuty → Slack → altura-alerts@.

5. API Standards & Performance Metrics

All APIs must follow these guidelines:

  • Use RESTful naming with snake_case only
  • Average response time < 300ms
  • Error rates < 0.5% weekly average

API quality scores are tracked in ApiHealthBoard and reviewed during sprint retros.

6. Secure Coding & Pre-Deploy Checklist

Security reviews are mandatory. Checklist includes:

  • No hardcoded secrets
  • Use altura-vault for secret management
  • Pass altura-secscan with score ≥ 95%

Final review by InfoSec is done in SecureSignPanel.

7. Legacy Systems & Refactor Policy

Legacy services must be tagged in Git as legacy_YYYYMM.

Refactor requests need a RefactorIntentForm and can’t exceed 10% of team sprint capacity without PM approval.
